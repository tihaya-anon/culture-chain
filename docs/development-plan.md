## 现阶段判断

当前最合适的阶段定义不是“完整 Phase 3 已完成”，而是：

```text
Phase A: Local Demo Ready
```

即：

- 合约可跑
- 前端可跑
- 本地交易链路可跑
- 但生产级基础设施未完成

## 建议的后续路线

### Phase B：链上闭环增强

目标：把当前 Demo 做到更稳定、更完整

- [ ] 修复本地链读写边界问题
- [ ] 增加更明确的交易成功 / 失败反馈
- [ ] 支持作品上架状态与购买后状态刷新
- [ ] 接入店铺注册前端流程
- [ ] 增加收益提取前端入口

### Phase C：数据层落地

目标：摆脱“链上直读 + mock fallback”的临时结构

- [ ] 设计 `/api/v1/works`、`/api/v1/profile` 等接口
- [ ] 接入 PostgreSQL + Prisma
- [ ] 实现链上事件同步到数据库
- [ ] 作品列表、详情、店铺页改为 API / DB 驱动

### Phase D：存储与内容发布

目标：让铸造流程从 demo metadata 升级到真实内容发布

- [ ] 接入真实 IPFS / Pinata 上传
- [ ] metadata 改为真实 URI
- [ ] 区分封面图与原始内容文件
- [ ] 为受限内容设计访问控制

### Phase E：测试网与上线准备

- [ ] Mumbai / Amoy 或目标测试网联调
- [ ] 移动钱包测试
- [ ] 合约安全复查
- [ ] 生产环境配置

---

## 当前优先级建议

如果继续沿着现有代码推进，建议优先顺序：

1. 店铺注册与店铺链上数据接入前端
2. 收益提取与交易状态完善
3. 数据库 / API 接入，去掉大量 mock fallback
4. 真实文件上传与 metadata 存储
5. 测试网联调
